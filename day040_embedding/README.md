# Day 40: 임베딩(Embedding)

## 개념 설명

임베딩(Embedding)은 Go 언어의 핵심적인 설계 개념으로, 타입(구조체 또는 인터페이스) 내에 다른 타입을 포함시켜 필드와 메서드를 재사용할 수 있게 해줍니다. 전통적인 객체 지향 언어의 상속(Inheritance)과는 달리, Go에서는 **컴포지션(Composition)**을 통해 코드 재사용을 구현합니다.

구조체 임베딩은 'is-a' 관계 대신 'has-a' 관계를 표현하는 데 적합합니다. 예를 들어, `Derived` 타입이 `Base`를 임베딩한다면 `Derived`는 `Base`의 필드와 메서드를 '가지고 있습니다'. 이는 더 유연한 설계를 가능하게 하며, 타입 간 결합도를 낮추는 효과가 있습니다.

메서드 프로모션(Method Promotion)은 임베딩된 타입의 메서드가 외부 타입의 메서드인 것처럼 호출될 수 있는 기능입니다. 이를 통해 `d.Base.Method()`가 아닌 `d.Method()`처럼 간결하게 호출할 수 있습니다.

인터페이스 임베딩은 여러 인터페이스를 조합하여 새로운 인터페이스를 정의할 때 사용됩니다. Go 1.14부터 이를 공식적으로 지원하며, 복잡한 인터페이스를 작은 단위로 나누어 재사용할 수 있게 해줍니다.

## 코드 설명

### 1. 구조체 임베딩
```go
type Derived struct {
	Base // 임베딩
	Name string
}
```
- `Derived`는 `Base`를 임베딩하여 `ID` 필드와 `Display()` 메서드를 직접 사용 가능합니다.
- `d.ID`와 `d.Display()`는 내부 `Base` 타입의 멤버인 것처럼 호출됩니다.
- `d.Base.ID`처럼 명시적으로 접근하는 것도 가능합니다.

### 2. 인터페이스 임베딩
```go
type ReadWriter interface {
	Reader
	Writer
}
```
- `ReadWriter`는 `Reader`와 `Writer` 인터페이스를 임베딩하여 두 인터페이스의 메서드를 모두 포함합니다.
- `MyReaderWriter`는 이 `ReadWriter` 인터페이스를 구현함으로써 `Read()`와 `Write()` 메서드를 모두 제공합니다.

### 3. 임베딩 vs 상속
- Go는 클래스 기반 상속을 지원하지 않으므로, 코드 중복 방지와 다형성을 위해 임베딩과 인터페이스를 조합하여 사용합니다.
- 이는 의존성 주입(DI)과 단위 테스트(Unit Test)를 작성할 때 더 유연한 아키텍처를 구축할 수 있게 합니다.

## 핵심 포인트
- **컴포지션 우선**: Go는 상속보다 컴포지션을 권장하며, 임베딩을 통해 'has-a' 관계를 구현합니다.
- **메서드 프로모션**: 임베딩된 타입의 메서드는 외부 타입의 메서드처럼 직접 호출할 수 있습니다.
- **인터페이스 조합**: 인터페이스 임베딩을 통해 여러 기능을 가진 인터페이스를 쉽게 확장할 수 있습니다.
- **명시적 접근**: `d.Field` 대신 `d.Base.Field`로 명시적으로 접근하여 의도를 명확히 할 수 있습니다.
- **유연한 설계**: 임베딩은 타입 간 느슨한 결합을 가능하게 하여 유지보수성과 테스트 용이성을 높입니다.
