# Day 43: atomic 연산

## 개념 설명
Go 언어에서 여러 고루틴(Goroutine)이 하나의 변수를 동시에 수정할 때 일반적인 mutex를 사용하면 성능 오버헤드가 발생할 수 있습니다. 특히 매우 빠른 속도로 카운터를 올리거나 상태를 확인하는 상황에서는 락(Lock)을 거는 것이 비효율적일 수 있습니다. 이때 사용하는 것이 `sync/atomic` 패키지입니다.

`atomic` 패키지는 CPU 하드웨어 수준에서 지원하는 원자성(Atomicity) 명령어를 이용합니다. 이는 연산 중 중단되지 않음을 보장하며, 락(Lock)을 사용하지 않고도 데이터 레이스(Race Condition)를 방지합니다. 이를 통해 락 기반 동기화보다 훨씬 빠른 속도로 공유 변수를 조작할 수 있습니다.

하지만 atomic 연산은 모든 타입에 적용 가능한 것은 아닙니다. `int64`, `uint32`, `Pointer`, `Value` 등 제한된 타입에 대해 각각의 함수가 제공됩니다. 또한, 복잡한 구조체 전체를 atomic하게 만드는 것은 불가능하며, 구조체 내부의 필드 단위로 접근해야 합니다.

## 코드 설명
제공된 예제 코드는 `sync/atomic` 패키지의 주요 함수들을 단계별로 보여줍니다.

1. **atomic.AddInt64**: `worker` 함수 내부에서 사용됩니다. 여러 고루틴이 동시에 호출하더라도 값을 정확히 1씩 증가시킵니다. mutex를 쓰지 않고도 5000이라는 정확한 값을 도출합니다.
2. **atomic.LoadInt64**: 현재 변수의 값을 안전하게 읽어옵니다. Write 연산이 진행 중인 중에 값을 읽어와도 안전하며, 최신 값을 보장합니다.
3. **atomic.StoreInt64**: 변수의 값을 특정 값(예: 0)으로 즉시 덮어씁니다. 이는 변수 초기화나 리셋 시 유용합니다.
4. **atomic.CompareAndSwapInt64 (CAS)**: "현재 값이 내가 기대한 값(oldVal)과 같은가? 그렇다면 newValue로 바꾼다"라는 로직입니다. 만약 값이 이미 바뀌었다면 연산을 하지 않고 false를 반환합니다. 이는 특정 상태 전환에만 연산을 적용할 때 매우 강력합니다.

## 핵심 포인트
*   **락(Lock) 없음**: `sync.Mutex`를 사용하지 않아 컨텍스트 스위칭 오버헤드가 없습니다.
*   **성능**: 하드웨어 명령어를 사용하므로 동기화 연산 중 가장 빠른 방식 중 하나입니다.
*   **타입 제한**: `int64`, `uint64`, `Pointer` 등 특정 타입 전용 함수가 존재합니다. (예: `AddInt32`, `LoadPointer`)
*   **Use Case**: 카운터, 플래그 변수, 상태 코드의 빠른 업데이트 등에 적합합니다.
