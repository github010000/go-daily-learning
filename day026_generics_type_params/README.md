# Day 26: 제네릭 — 타입 파라미터

## 개념 설명
Go 1.18부터 도입된 제네릭(Generics)은 타입을 파라미터로 받아서 타입 안전성을 유지하면서도 중복 코드를 줄이는 문법입니다. `func F[T any](x T)`에서 `[T any]`는 타입 파라미터 선언부이며, `any`는 `interface{}`의 별칭으로 모든 타입을 허용하는 가장 느슨한 제약 조건입니다. 타입 파라미터를 사용하면 슬라이드, 맵, 함수 등 여러 타입에 공통으로 적용되는 로직을 한 번에 정의할 수 있어 코드의 재사용성과 가독성이 크게 향상됩니다.

타입 제약(Type Constraints)은 타입 파라미터가 허용할 수 있는 타입의 범위를 제한하는 규칙입니다. 인터페이스 리터럴을 사용하여 `~int | ~float64`와 같이 허용할 타입을 명시할 수 있으며, `~` 기호는 해당 타입 자체와 그 underlying type(하위 타입)을 모두 허용합니다. 특히 `comparable` 내장 제약은 `==`나 `!=`와 같은 비교 연산이 가능한 타입에만 함수를 제한하여, 슬라이드 탐색이나 맵 키 사용 시 런타임 panic을 사전에 방지합니다.

제네릭과 인터페이스 중 무엇을 선택할지는 문제의 성질에 따라 달라집니다. 인터페이스는 "어떤 행동을 하는가"에 초점을 맞춰 런타임 다형성을 제공하며, 타입이 유동적인 경우에 적합합니다. 반면 제네릭은 "어떤 타입인가"에 초점을 맞춰 컴파일 타임 타입 안전성과 성능 최적화(리플렉션 제거, 인라인 최적화)를 제공합니다. 일반적으로 타입이 명확하고 특정 연산이나 비교가 필요한 경우 제네릭을, 동작만 규정하고 구현을 유보해야 할 경우 인터페이스를 사용하는 것이 Go의 모범 사례입니다.

## 코드 설명
- `PrintAny[T any]`: `any` 제약을 사용해 모든 타입을 처리하는 기본 제네릭 함수입니다. `%T` 포맷을 사용해 실제 런타임 타입을 확인하며, 타입 파라미터는 호출 시 자동으로 추론됩니다.
- `Sum[T Number]`: 커스텀 타입 제약 `Number`를 적용해 정수 및 실수 타입에만 동작하도록 제한합니다. `~` 기호를 사용해 `int`, `int64`, `float64`의 원시 타입과 하위 타입을 모두 허용합니다.
- `FindIndex[T comparable]`: `comparable` 제약을 사용해 슬라이드 탐색 시 비교 연산이 안전한 타입만 허용합니다. 만약 `comparable`이 아닌 타입(예: 슬라이드, 맵, 함수)을 전달하면 컴파일 단계에서 에러가 발생합니다.
- `GenericDescribe[T Stringer]`: 인터페이스를 제약 조건으로 사용하는 예시입니다. 제네릭에서 인터페이스는 타입이 해당 인터페이스를 만족한다는 것을 명시하는 용도로 사용되며, 타입 안전성과 코드 명확성을 높여줍니다.

## 핵심 포인트
- `any`는 `interface{}`의 별칭으로, 모든 타입을 허용하는 기본 제네릭 제약입니다.
- 타입 제약은 인터페이스 리터럴(`type T interface { ~int | ~string }`)로 정의하며, `~`는 하위 타입 허용을 의미합니다.
- `comparable`은 비교 연산(`==`, `!=`)이 가능한 타입에만 적용할 수 있는 내장 제약으로, 런타임 오류를 예방합니다.
- 제네릭은 컴파일 타임 타입 검증과 성능 최적화에, 인터페이스는 런타임 유연성과 다형성에 강점이 있습니다.
- 타입 파라미터는 함수 시그니처에 `[T Constraint]` 형태로 선언되며, 호출 시 타입 추론이 자동 적용되어 구문이 간결해집니다.
