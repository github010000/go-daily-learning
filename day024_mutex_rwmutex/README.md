# Day 24: Mutex와 RWMutex

## 개념 설명

Go 언어에서 여러 고루틴이 동일한 메모리 공간(변수, 맵, 구조체 등)에 동시에 접근하여 값을 수정하려고 할 때 **데이터 경합(Race Condition)**이 발생합니다. 이를 방지하기 위해 동기화 도구가 필요한데, 그 중 가장 대표적인 것이 `sync.Mutex`와 `sync.RWMutex`입니다.

`sync.Mutex`는 **Mutual Exclusion(상호 배제)**의 약자로, 한 번에 단 하나의 고루틴만이 임계 구역(Critical Section)에 접근할 수 있도록 보장합니다. 즉, 한 고루틴이 `Lock()`을 호출하면 다른 모든 고루틴은 `Unlock()`이 호출될 때까지 대기 상태가 됩니다.

`sync.RWMutex`는 Mutex의 최적화 버전입니다. 데이터의 **읽기 작업은 빈번하지만 쓰기 작업은 드문 경우**에 매우 효율적입니다. `RLock()`을 사용하면 여러 고루틴이 동시에 데이터를 읽을 수 있도록 허용하면서도, `Lock()`(쓰기 작업)이 발생할 때는 모든 읽기/쓰기 작업을 차단하여 데이터의 일관성을 유지합니다.

## 코드 설명

1.  **SafeCounter (sync.Mutex 사용)**:
    *   `mu sync.Mutex`를 구조체 내부에 포함합니다.
    *   `Inc()` 메서드에서 `c.mu.Lock()`을 통해 다른 고루틴의 접근을 막고, `defer c.mu.Unlock()`을 사용하여 작업 완료 후 반드시 잠금을 해제합니다.
    *   `defer`를 사용하는 이유는 함수 중간에 패닉이 발생하거나 여러 개의 return 문이 있더라도 잠금이 해제되지 않아 발생하는 **데드락(Deadlock)**을 방지하기 위함입니다.

2.  **ConfigMap (sync.RWMutex 사용)**:
    *   `mu sync.RWMutex`를 사용하여 읽기와 쓰기 권한을 분리합니다.
    *   `Set()` 메서드(쓰기): `Lock()`을 사용하여 새로운 데이터가 들어올 때 다른 모든 고루틴(읽기 및 쓰기 모두)의 접근을 차단합니다.
    *   `Get()` 메서드(읽기): `RLock()`을 사용하여 다른 읽기 고루틴들이 동시에 데이터를 읽을 수 있도록 허용합니다. 단, 쓰기 작업(`Lock`)이 진행 중일 때는 대기합니다.

3.  **main 함수**:
    *   `sync.WaitGroup`을 사용하여 모든 고루틴이 작업을 마칠 때까지 메인 함수가 종료되지 않도록 제어합니다.
    *   1,000번의 동시 증가 작업을 통해 Mutex가 데이터 경합을 어떻게 막는지 확인합니다.
    *   동시 읽기/쓰기 시나리오를 통해 RWMutex의 동작을 시뮬레이션합니다.

## 핵심 포인트

*   **Lock/Unlock 쌍 맞추기**: `Lock()`을 호출했다면 반드시 `Unlock()`을 호출해야 합니다. `defer`를 사용하는 것이 가장 안전한 관습입니다.
*   **Race Condition 방지**: 공유 자원에 대한 수정이 일어나는 모든 경로에는 반드시 잠금 메커니즘이 적용되어야 합니다.
*   **RWMutex의 효율성**: 읽기 작업이 압도적으로 많은 경우 `sync.Mutex` 대신 `sync.RWMutex`를 사용하여 성능(Concurrency)을 높일 수 있습니다.
*   **데드락(Deadlock) 주의**: 이미 잠긴 뮤텍스를 다시 `Lock()`하려고 하거나, 잠금을 해제하지 않은 상태로 무한 대기 상태에 빠지는 것을 주의해야 합니다.
