# Day 20: 에러 처리 (Error Handling)

## 개념 설명
Go 언어는 다른 언어(Java, Python 등)에서 사용하는 `try-catch` 문법을 통한 예외 처리 방식을 사용하지 않습니다. 대신, 함수가 에러를 **값(value)**으로서 반환하도록 설계되어 있습니다. 이를 통해 에러 처리를 프로그램의 흐름 제어의 일부로 명시적으로 다룰 수 있게 합니다.

Go의 모든 에러는 `error`라는 내장 인터페이스를 구현합니다. `error` 인터페이스는 단 하나의 메서드인 `Error() string`만을 가집니다. 이 구조 덕분에 개발자는 단순한 문자열 에러부터, 풍부한 컨텍스트를 가진 복잡한 구조체 에러까지 일관된 방식으로 처리할 수 있습니다.

최신 Go 버전(1.13 이상)에서는 에러 래핑(Error Wrapping) 기능이 매우 중요해졌습니다. 에러가 발생했을 때 단순히 에러를 전달하는 것이 아니라, `fmt.Errorf`와 `%w`를 사용하여 "어떤 맥락에서 이 에러가 발생했는지"를 기존 에러 위에 덧씌워 전달할 수 있습니다. 이를 통해 에러의 추적성(Traceability)을 높일 수 있습니다.

## 코드 설명
- **`errors.New`**: 가장 단순한 형태의 에러를 생성합니다. 고정된 메시지를 가진 에러 상수를 정의할 때 주로 사용합니다.
- **`ValidationError` 구조체**: `error` 인터페이스를 구현하기 위해 `Error() string` 메서드를 구현한 커스텀 에러 타입입니다. 에러가 발생한 필드(`Field`)와 메시지(`Message`)라는 추가 데이터를 포함합니다.
- **에러 래핑 (`%w`)**: `findUserByID` 함수 내에서 `fmt.Errorf("... %w", ErrUserNotFound)`를 사용하여 기존의 `ErrUserNotFound` 에러를 감싸 새로운 에러를 생성합니다. 이는 에러의 계층 구조를 유지합니다.
- **`errors.Is`**: 에러 체인(Error Chain)을 따라 내려가며, 특정 에러 값이 포함되어 있는지 확인합니다. 래핑된 에러를 검사할 때 필수적입니다.
- **`errors.As`**: 에러 체인을 조사하여 에러가 특정 타입(예: `*ValidationError`)으로 변환 가능한지 확인하고, 변환이 가능하다면 해당 타입으로 포인터를 채워줍니다. 이를 통해 커스텀 에러의 필드에 접근할 수 있습니다.

## 핵심 포인트
- **에러는 값이다**: 에러를 `if err != nil` 패턴으로 명시적으로 체크하는 것이 Go의 관례입니다.
- **에러 래핑 (`%w`)**: 에러의 맥락을 유지하기 위해 기존 에러를 버리지 말고 `%w`로 감싸서 전달하세요.
- **`errors.Is` vs `errors.As`**: 
    - 특정 에러 **값**과 일치하는지 확인할 때는 `Is`.
    - 특정 에러 **타입**으로 변환하여 내부 데이터에 접근하고 싶을 때는 `As`.
- **커스텀 에러 활용**: 에러 발생 지점의 메타데이터(에러 코드, 발생 필드 등)가 필요하다면 구조체를 활용한 커스텀 에러를 구현하세요.
